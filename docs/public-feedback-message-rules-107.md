# GENESIS — Public Feedback Message Rules 107

## Purpose

Define the writing rules for public feedback drafts.

These rules prevent the launch content from sounding robotic, salesy, or spammy.

## Voice

Use:

- human founder voice
- simple technical language
- honest uncertainty
- direct question
- short paragraphs
- one idea per paragraph

Avoid:

- corporate language
- hype
- exaggerated claims
- AI buzzword overload
- long product explanations
- "revolutionary"
- "game-changing"
- "autonomous enterprise platform"
- "10x everything"
- repeated calls to action

## Best Opening Pattern

Start with a real problem:

- I hit this problem while building with AI agents.
- One of my AI agents broke something quietly.
- I kept seeing AI-generated code move faster than review.
- I wanted a small checkpoint before generated code gets executed.

Do not start with:

- We are excited to announce.
- Introducing GENESIS.
- The future of AI development is here.
- I built the ultimate AI safety platform.

## Product Description Rule

Use one simple sentence:

GENESIS is a small safety gate for AI-generated code changes.

Optional second sentence:

It classifies changes as ALLOW, REQUIRE_APPROVAL, or BLOCK before they move forward.

Do not over-explain.

## Demo Link Rule

Use the demo link once:

https://aicodesafety.com

Use the public repo link only where appropriate:

https://github.com/aicodesafety/genesis-ai-code-safety-demo

Do not overload every post with too many links.

## Feedback Question Rule

Every draft must end with one clear question.

Good questions:

- Where should this gate live: CLI, PR, CI, editor, or agent runtime?
- Have you seen AI-generated code break something unexpectedly?
- Does this feel like a real workflow problem or just a demo curiosity?
- What would make this useful in a real team workflow?

Bad questions:

- Would you buy this?
- Who wants access?
- Should I launch this?
- Can you share this?
- Would you pay for this?

## Payment Rule

Do not mention:

- price
- checkout
- Lemon Squeezy
- Buy Now
- paid pilot
- payment link

Exception:

Only after a person asks about setup, pilot, or cost.

## Private Core Rule

Do not describe private internals.

Do not reveal adapters, internal execution systems, or private repo details.

Do not say the public demo is the full system.

Allowed:

This is a small public demo shell. The private core is not included.

## Claims Rule

Allowed:

- small demo
- early validation
- safety gate idea
- pre-execution checkpoint
- AI-generated code risk
- ALLOW / REQUIRE_APPROVAL / BLOCK classification

Not allowed:

- prevents all AI bugs
- enterprise-ready
- production-grade platform
- replaces tests
- replaces code review
- supports every stack
- fully autonomous safety
- guaranteed protection

## Platform-Specific Tone

### GitHub Discussion

Tone:

technical, direct, structured.

Best format:

question + context + options.

### LinkedIn

Tone:

personal founder story.

Best format:

problem story + what I built + feedback question.

### X / Twitter

Tone:

short, sharp, conversational.

Best format:

mini-thread or single post.

### Show HN

Tone:

plain, technical, no hype.

Best format:

what I built + why + links + question.

### Reddit

Tone:

community-first, question-first.

Best format:

ask the workflow question before posting the link.

### Dev.to

Tone:

technical build log.

Best format:

problem, simple architecture, demo, lessons.

### Medium

Tone:

longer essay.

Best format:

story, insight, product direction, reflection.

## Anti-Spam Rules

Do not:

- post the same text everywhere
- publish all channels at once
- DM people from public comments
- repost aggressively
- argue with criticism
- add links where not allowed
- pretend to be a large company
- hide that this is early
- overstate adoption

## Criticism Handling

If someone criticizes the idea:

Respond with curiosity.

Good reply:

That is useful feedback. Where do you think the right checkpoint should be, if anywhere?

Bad reply:

You do not understand the product.

## Final Rule

The goal is not to look big.

The goal is to sound real, technical, careful, and worth replying to.
