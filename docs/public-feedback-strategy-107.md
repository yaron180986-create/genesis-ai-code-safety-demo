# GENESIS — Public Feedback Strategy 107

## Purpose

Define the first public feedback strategy for GENESIS / AI Code Safety.

This is a PREPARE_ONLY document.

No public post is authorized by this document.

No platform action is authorized by this document.

## Core Goal

Turn the existing live demo into real feedback from developers who use AI coding tools.

The goal is not traffic.

The goal is signal.

## Primary Validation Question

Does a pre-execution safety gate for AI-generated code feel like a real workflow problem?

## Product Context

GENESIS is an AI Code Safety Gate.

It classifies AI-generated code changes as:

- ALLOW
- REQUIRE_APPROVAL
- BLOCK

Current public demo:

https://aicodesafety.com

Public repo:

https://github.com/aicodesafety/genesis-ai-code-safety-demo

## Current Stage

GENESIS is in early public validation.

The correct tone is:

- honest
- technical
- human
- humble
- founder-led
- feedback-seeking
- not salesy
- not corporate
- not growth-hacky

## What We Are Testing

We are testing:

1. Whether developers recognize the AI-generated code safety problem.
2. Where they think the safety gate should live.
3. Whether ALLOW / REQUIRE_APPROVAL / BLOCK is understandable.
4. Whether the demo communicates the problem quickly.
5. Whether anyone asks for CI, GitHub Action, CLI, PR scanner, or editor integration.
6. Whether anyone describes real workflow pain.

## What We Are Not Testing Yet

We are not testing:

- pricing
- checkout conversion
- Product Hunt launch
- enterprise sales
- automated outreach
- mass adoption
- viral content
- paid ads
- full platform integration
- private GENESIS core

## Channel Priority

### 1. GitHub Discussion

Reason:

Owned context.

Low risk.

Good place to ask where the safety gate should live.

Primary question:

Where should an AI code safety gate live?

### 2. LinkedIn

Reason:

Founder story works well.

Good for personal credibility.

Good for non-spammy feedback request.

Primary angle:

An AI agent broke something quietly, so I built a small safety gate demo.

### 3. X / Twitter

Reason:

Short technical story.

Good for lightweight feedback.

Primary angle:

AI coding needs safety gates, not only better generation.

### 4. Show HN

Reason:

Strong technical feedback.

High signal if the post lands.

Higher criticism risk.

Primary angle:

Show HN: I built a small safety gate for AI-generated code.

### 5. Reddit

Reason:

Strong communities but high self-promotion sensitivity.

Should start question-first.

Primary angle:

Where should a checkpoint live for AI-generated code?

### 6. Dev.to

Reason:

Good for technical build log.

Slower channel.

Primary angle:

I built a small safety gate after AI-generated code broke a working module.

### 7. Medium

Reason:

Good for longer founder essay.

Slower channel.

Primary angle:

AI-generated code does not only need better generation. It needs safety gates.

## Success Signals

Strong signals:

- "This happened to me."
- "Can this run in CI?"
- "Can this be a GitHub Action?"
- "Can it scan PRs?"
- "Can this work with Cursor?"
- "Can this run locally?"
- "Can we test it?"
- "This is a real issue in our workflow."
- "We need something like this."

Weak signals:

- likes
- stars only
- generic praise
- "cool idea"
- "nice"
- no workflow pain
- curiosity without use case

## Response Classification

Use:

- GHOST
- WEAK_SIGNAL
- PAIN_CONFIRMED
- INTEGRATION_INTEREST
- DEMO_REQUEST
- PILOT_CANDIDATE
- NOT_RELEVANT

## Payment Boundary

No Lemon Squeezy.

No checkout.

No Buy Now.

No payment link.

No pricing pitch.

Payment can be discussed only after:

- pain is confirmed
- pilot candidate exists
- user approves pilot qualification
- user approves payment flow separately

## Private Core Boundary

Do not describe private GENESIS internals.

Do not mention private repo.

Do not imply the public demo includes the private core.

Allowed phrase:

This is a public demo shell. The private GENESIS core is not included.

## Decision Rule

If public feedback produces no real pain signal:

Improve positioning and targeting.

Do not build more features.

If feedback produces integration interest:

Prepare integration discovery.

If feedback produces a pilot candidate:

Prepare pilot qualification.

## Final Rule

GENESIS should go to market through trust, clarity, and real technical feedback — not spam.
