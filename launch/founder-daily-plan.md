# Founder Daily Plan — GENESIS AI Code Safety Gate

> A launch is not one big scary event. It is a sequence of small controlled exposures.

---

## Core Rules

**Done is enough.**
Each day has a minimum version. Doing the minimum is a successful day. You do not need to do everything on the list. You need to do one thing per day.

**You are not asking for approval. You are running an experiment.**
Every outreach message is a data collection request. Every post is a hypothesis test. The outcome does not define the product. It defines the message.

**You control the pace.**
There is no external deadline. If you need to extend Day 2 into two days — extend it. The plan is a template, not a contract.

---

## Day 1 — Polish and First Contact

**Goal:** Make sure the demo is clean and send it to 3 trusted people.

**Minimum version (do this):**
- Open https://aicodesafety.com and test all three demo scenarios
- Fix anything that looks broken or unclear
- Send to 1 trusted developer with this message: "I built something — does the problem feel real to you?"

**Normal version:**
- Test the demo
- Check the README for any outdated text
- Add GitHub repo topics (see communities.md) if not already set
- Send to 3 trusted developers using the outreach-messages.md drafts
- Record who you sent to in launch-tracking.md

**Ambitious version:**
- All of the above
- Review the demo one more time from the perspective of someone who has never seen it
- Make a note of anything confusing for the next iteration
- Send to 5 people

**Anxiety-safe version:**
- Just test the live demo and send one message.
- That is enough for Day 1.

---

## Day 2 — Expand Outreach

**Goal:** Send to 7 more people across different relationship types.

**Minimum version (do this):**
- Send to 3 more people using the outreach-messages.md templates
- Reply to any responses from Day 1

**Normal version:**
- Send to 7 people covering: senior engineer, startup founder, security engineer, AI tools builder
- Use the Hebrew version for anyone in your Hebrew-speaking network
- Log each outreach in launch-tracking.md
- Reply personally to every Day 1 response

**Ambitious version:**
- All of the above
- Write a short note to yourself about what Day 1 responses revealed
- If any response suggested a positioning confusion, update the pitch framing in your notes before Day 3

**Anxiety-safe version:**
- Send to 3 people only.
- Do not read responses obsessively. Check once in the evening.

---

## Day 3 — First LinkedIn Post

**Goal:** Make the project visible to your professional network.

**Minimum version (do this):**
- Post the "short version" LinkedIn draft from social-posts.md
- Walk away. Do not check comments for at least 2 hours.

**Normal version:**
- Post the "founder story" or "technical" LinkedIn version
- Engage with any comments by end of day
- Do not post again on Day 3 — one post is enough

**Ambitious version:**
- Post LinkedIn
- Reply to all comments with substance, not just thanks
- Note any patterns in how people respond (confusion, engagement, critique)

**Anxiety-safe version:**
- Copy the short version LinkedIn post. Post it without editing.
- You do not need to optimize. You need to post.
- If it feels too hard: post the 3-sentence version only.

**What to do if nobody engages:**
- Nothing. Wait 24 hours.
- Check GitHub traffic — real views don't always produce comments.
- Proceed to Day 4 regardless.

---

## Day 4 — Technical Community Post

**Goal:** Reach developers who are actively working with AI coding tools.

**Minimum version (do this):**
- Post to one subreddit or Hacker News Show HN using the drafts from social-posts.md

**Normal version:**
- Post Show HN on Hacker News (morning US Eastern time for best reach)
- OR post to r/programming if Hacker News feels too exposed
- Write the OP comment immediately after posting (use the HN comment draft from social-posts.md)
- Reply to every comment within 3 hours

**Ambitious version:**
- Post to Hacker News
- Post to r/programming the same day (different framing)
- Monitor and respond throughout the day

**Anxiety-safe version:**
- Post to one Reddit community only. r/programming is the lowest risk.
- Use the exact draft from social-posts.md. Do not rewrite it from scratch.

**What to do if you get critical comments:**
- Write down the criticism before responding.
- Reply to technical criticism specifically and without defensiveness.
- Do not delete the post.
- Critical comments are valuable — they tell you exactly where the positioning gaps are.

---

## Day 5 — Collect Feedback

**Goal:** No new posts. Process what you have received.

**Minimum version (do this):**
- Read all responses and replies from Days 1–4
- Write one sentence summarizing what you learned

**Normal version:**
- Read all responses
- Fill in the feedback tracker table in launch-tracking.md
- Identify the 1–2 most important signals (positive or critical)
- Note any patterns (recurring confusion, recurring question, recurring compliment)
- Reply to any unanswered comments

**Ambitious version:**
- All of the above
- Write a short paragraph on what the first 4 days revealed about positioning, target audience, and message clarity
- Identify one specific thing to improve before Day 6

**Anxiety-safe version:**
- Skim the responses. Write one sentence about what you noticed. Close the laptop.
- That is enough for Day 5.

---

## Day 6 — Improve One Thing

**Goal:** Make one improvement based on what you learned.

**Minimum version (do this):**
- Update one sentence in the README based on a confusion pattern you noticed

**Normal version:**
- Update the README intro or demo description based on feedback
- If positioning confusion came up: update the one-line pitch on the repo or landing page
- Post a short LinkedIn follow-up (2–3 sentences): "Launched this earlier this week. Here's what I learned."

**Ambitious version:**
- Update README and demo messaging
- Write a short Dev.to article using the outline from social-posts.md
- Post the Dev.to article

**Anxiety-safe version:**
- Make one single small edit to the README.
- That counts as a successful Day 6.

---

## Day 7 — Decide Next Move

**Goal:** Evaluate the week and decide what comes next.

**Minimum version (do this):**
- Look at the launch-tracking.md metrics table you filled in
- Write one paragraph: what worked, what didn't, what you'll do differently

**Normal version:**
- Review all metrics
- Use the decision table in launch-tracking.md to determine the recommended next action
- Choose one of: continue current approach / refine message / refine demo / change target audience / pause
- Write a short note to your future self about what you know now that you didn't know on Day 1

**Ambitious version:**
- Full retrospective
- Write a short update post on LinkedIn: "One week after first launch — here's what I learned"
- Plan the next 7 days based on the decision table

**Anxiety-safe version:**
- Just look at the GitHub traffic numbers.
- If there are views, people found it. That is real.
- Decide tomorrow whether to continue.

---

## End of Week — Reminders

- You shipped something real.
- You exposed it to real people.
- That is not nothing. That is the whole point of Week 1.

The next decision is not "did I succeed?" The next decision is "what signal do I have, and what does it tell me to do next?"

That is how products get built.
